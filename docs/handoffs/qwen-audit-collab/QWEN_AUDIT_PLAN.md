# Qwen Audit Plan

【签名】Qwen
【时间】2026-04-11
【设备】MacBook

## 1. 方案目标

本整改方案旨在系统性解决Qwen审计报告中识别的安全风险，同时确保：

1. **风险缓解**：优先处理P1级别高风险问题，降低攻击面
2. **服务隔离**：按服务边界组织修复工作，最小化跨服务影响
3. **可持续性**：建立安全开发实践，防止同类问题复发
4. **渐进实施**：分批次推进，每批完成后进行安全验证

**核心原则**：
- 只读审查发现的问题必须通过正式任务流程修复
- 跨服务改动需要架构师预审
- 高风险修复需要充分测试验证

## 2. 建议处理顺序

按风险级别和修复复杂性排序：

### 第一阶段：立即行动（1-2周）
1. **P1-专项复核/加固准备**：回测表达式执行链安全加固（F-001）
2. **模式风险处理**：data侧命令执行与查询构造可达性复核（F-002、F-003）

### 第二阶段：核心加固（2-4周）
3. **依赖管理**：第三方库安全审查与部署治理复查

## 3. 建议批次拆分

### 批次A：表达式执行专项复核 / 加固准备
- **批次名**：BACKTEST-EXPRESSION-SECURITY-ASSESSMENT
- **目标问题编号**：F-001
- **建议归属服务**：backtest服务
- **是否跨服务**：否（仅backtest服务内部）
- **建议文件范围**：
  - `services/backtest/src/backtest/generic_strategy.py`
- **验证方式**：
  1. 代码审查：复核`_SafeExpressionValidator`实现
  2. 代码审查：复核`_ALLOWED_FUNCTIONS`白名单
  3. 安全测试：评估表达式复杂度/深度限制需求
  4. 沙箱方案评估：调研替代`eval()`的安全方案
- **主要风险**：
  - 策略计算准确性：安全措施可能影响表达式求值
  - 性能影响：沙箱环境可能增加开销
  - 跨版本兼容性：可能影响已有策略文件
- **批次类型**：**只读复查批次** - 本批次目标是评估和规划，而非直接修复

### 批次B：data命令执行与查询构造可达性复核
- **批次名**：DATA-COMMAND-QUERY-REACHABILITY-ASSESSMENT
- **目标问题编号**：F-002、F-003
- **建议归属服务**：data服务
- **是否跨服务**：否（仅data服务内部）
- **建议文件范围**：
  - `services/data/src/scheduler/data_scheduler.py`
  - `services/data/src/ops/backup.py`
  - `services/data/src/collectors/overseas_minute_collector.py`
  - `services/data/src/data/storage.py`
- **验证方式**：
  1. 数据流分析：确定哪些参数来自固定配置，哪些存在动态输入面
  2. 调用链分析：验证`sql_where`/`order_by`是否存在真实调用链
  3. 安全评估：区分Bandit/模式级告警与真实漏洞风险点
- **主要风险**：
  - 识别准确度：需要精确区分模式风险与真实漏洞
- **批次类型**：**只读复查批次** - 本批次目标是证据补强与风险评估，而非直接修复

### 批次C：可建单修复批次（基于前两批评估结果）
- **批次名**：BACKTEST-EXPRESSION-SECURITY-IMPLEMENTATION
- **目标问题编号**：F-001（基于批次A评估结果）
- **建议归属服务**：backtest服务
- **是否跨服务**：否（仅backtest服务内部）
- **建议文件范围**：
  - `services/backtest/src/backtest/generic_strategy.py`
- **验证方式**：
  1. 安全测试：恶意表达式注入测试
  2. 功能回归测试：确保策略计算正确性
  3. 性能基准测试：验证安全措施不影响性能
- **主要风险**：
  - 策略计算准确性：安全措施可能影响表达式求值
  - 性能影响：沙箱环境可能增加开销
- **批次类型**：**可建单候选批次** - 基于批次A评估结果后可正式建单修复

### 批次D：依赖与部署治理只读复查
- **批次名**：DEPS-DEPLOY-GOVERNANCE-ASSESSMENT
- **目标问题编号**：依赖安全与部署配置复查
- **建议归属服务**：所有服务
- **是否跨服务**：是（影响所有服务）
- **建议文件范围**：
  - 项目根目录：`requirements.txt`、`pyproject.toml`
  - 部署配置：`docker-compose.*.yml`（只读审查，非直接修改）
  - 依赖清单：所有`requirements*.txt`文件
- **验证方式**：
  1. 依赖安全扫描：使用Safety、Trivy等工具
  2. 配置审计：只读审查部署模板安全配置
- **主要风险**：
  - **需项目架构师预审 + Token**：若后续需触及真实部署文件
- **批次类型**：**只读复查批次** - 本批次仅作评估，不直接修改受保护文件

## 4. 必须先预审的事项

以下事项需要在任何修复批次开始前由架构师预审：

1. **共享契约修改**：任何涉及`shared/contracts/`的改动
2. **治理模块变更**：`governance/`目录下的任何修改
3. **跨服务API变更**：服务间接口协议修改
4. **数据库模式变更**：可能影响数据持久化的模式改动
5. **认证授权架构**：任何身份验证或授权逻辑修改

**预审检查清单**：
- [ ] 是否影响现有生产环境？
- [ ] 是否有回滚计划？
- [ ] 是否已评估所有依赖影响？
- [ ] 是否已与相关服务负责人沟通？
- [ ] 是否已创建相应的测试用例？

## 5. 可以只读继续确认的事项

以下问题需要进一步只读审查，无需立即修复：

1. **SimNow网关内存安全**：进一步审查`services/sim-trading/src/api/router.py`中凭证的内存处理
   - **审查方法**：代码分析、内存安全模式检查
   - **预期产出**：内存安全最佳实践文档
   - **时间估算**：2-3人日

2. **治理token密码学实现**：深入审查`governance/jbt_lockctl.py`密码学实现
   - **审查方法**：密码学专家代码审查、实现对比分析
   - **预期产出**：密码学实现评估报告
   - **时间估算**：3-5人日

3. **服务间通信安全**：审查服务间数据流的安全边界
   - **审查方法**：架构分析、数据流图绘制、安全边界验证
   - **预期产出**：服务间通信安全规范
   - **时间估算**：5-7人日

## 6. 不建议现在动的事项

以下事项不建议在当前阶段修改，原因如下：

1. **第三方库内部实现**：如`aiohttp/cookiejar.py`中的pickle使用
   - **理由**：属于第三方库内部实现，除非发现具体漏洞，否则不应修改
   - **建议**：监控库的安全公告，及时更新版本
   - **风险**：修改第三方库会导致维护困难和兼容性问题

2. **遗留代码重构**：非安全相关的代码质量改进
   - **理由**：当前焦点是安全风险，非安全重构可能分散资源
   - **建议**：在专门的质量改进批次中处理
   - **风险**：重构可能引入新的安全漏洞

3. **性能优化**：非安全相关的性能改进
   - **理由**：性能优化应在安全加固完成后进行
   - **建议**：建立性能基准，在安全批次后专门优化
   - **风险**：过早优化可能掩盖安全需求

4. **UI/UX改进**：前端界面和用户体验改进
   - **理由**：与核心安全目标无关
   - **建议**：由前端团队在独立迭代中处理
   - **风险**：界面改动可能影响用户工作流程

**总体建议**：遵循"安全第一、稳定第二、功能第三"的原则，当前阶段集中资源解决已识别的安全风险，避免范围蔓延。优先完成只读复查批次(A、B、D)，然后基于评估结果推进可建单修复批次(C)。

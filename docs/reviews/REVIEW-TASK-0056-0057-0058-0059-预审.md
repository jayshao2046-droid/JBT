### 架构师确认点回答

**Q1: sandbox_engine.py 取数通过 FactorLoader._fetch_bars() 还是重新 httpx 调用 data API？**

**裁定：通过 FactorLoader 内部方法。** 理由：
- `factor_loader.py` 已存在于同目录（`research/`），内部已有 `_fetch_bars()` 通过 httpx 调用 data API `/api/v1/bars`
- 沙箱属 decision 内部研究工具，复用同目录已有的 FactorLoader 取数链路最简单
- 不新建独立的 data client，避免重复代码

**Q2: SandboxResult 是否需登记到 shared/contracts？**

**裁定：NO，不需要。** 理由：
- SandboxResult 仅在 decision 内部使用（research → sandbox route）
- 不跨服务传输，不需要契约登记
- 如未来 backtest 需要消费沙箱结果，届时再登记

### ⚠️ 规格书注意事项

规格书 §3.3 定义了 `FuturesSandbox` 类名，但 TASK-0057（CB2'）设计约束要求"**不新建独立 StockSandbox 类，通过 asset_type 扩展**"。建议实施时：
- 类名使用 `SandboxEngine`（通用），而非 `FuturesSandbox`（特化）
- 构造时传入 `asset_type: Literal["futures", "stock"]`
- 此项为**实施建议**，不影响预审通过

### 综合结论：✅ 预审通过

### 架构师确认点回答

**Q1: A股 T+1 约束实现粒度？**

**裁定：sandbox 级别模拟即可。** 理由：
- 沙箱是快速回测工具，不涉及实盘风控层
- T+1 在回测循环中通过「买入日标记 → 次日才可卖出」逻辑实现
- sim-trading 的真实风控（实盘保护）是另一层，不在此任务范围

**Q2: 涨跌停价格从哪取？**

**裁定：从 bars 数据的 `high_limit` / `low_limit` 字段读取。** 理由：
- data API 的 stock bars 应包含涨跌停价格字段
- 不依赖实时价推送（沙箱用历史数据回测）
- 如 data bars 缺少涨跌停字段，需在 C0-1（TASK-0050）或 C0-2（TASK-0053）中补充

规格书 §2 中 `sandbox_engine.py` 同时出现在"修改"和"planned-placeholder"两个节——技术上不矛盾（planned-placeholder 说明激活前依赖，修改说明本任务操作），但实施时需注意：CB2' 激活前验证 sandbox_engine.py 确实已由 CA2' 产出。

| 前置依赖 | ✅ 合理 | TASK-0055（CG2，人工审核通过后才允许信号进 sim-trading）|

### ⚠️ 路由前缀差异发现

**重要发现：** Decision 服务当前**不使用 `/api/v1` 全局前缀**。

| 服务 | 路由前缀模式 | 示例 |
|------|-------------|------|
| decision | 短前缀 `/signals`、`/approvals`、`/strategies` | `POST /signals/dispatch` |
| sim-trading | `/api/v1` 全局前缀 | `POST /api/v1/signals/receive` |

规格书 §3.1 写的「Decision 侧: `/api/v1/signals/dispatch`」不准确，实际路径为 `/signals/dispatch`。

**裁定：** 这不影响功能——`signal_dispatcher.py` 调用 sim-trading 时使用 `{SIM_URL}/api/v1/signals/receive` ✅。但规格书的路径描述需在实施时修正，避免混淆。此项不阻塞预审。

### 架构师确认点回答

**Q1: sim-trading 是否已有订单去重机制？CA6 是否需新建 signal_id 幂等表？**

**裁定：需新建简单幂等缓存。** 理由：
- sim-trading 当前 router.py 无信号相关端点，更无幂等机制
- 网络重试可能导致重复信号送达
- 建议在 router.py 内用内存 dict/set 做 `signal_id` 幂等（TTL 24h），不新建独立文件
- 持久化幂等表（DB 级）留待后续需求

**Q2: signal_dispatcher.py 网络异常时的重试策略？**

**裁定：同步重试（tenacity）。** 理由：
- 信号派发是同步请求-响应模式
- tenacity 已是 Python 生态标准重试库，无额外依赖
- 异步队列（Redis/RabbitMQ）过重，当前 decision 不依赖消息队列
- 重试参数建议：max_attempts=3, wait_exponential(min=1, max=10)

**Q3: 跨服务信号格式是否需登记到 shared/contracts？**

**裁定：YES，需要。** 理由：
- 信号请求体（signal_id, strategy_id, symbol, direction, quantity, price...）是 decision ↔ sim-trading 的跨服务契约
- 必须登记到 `shared/contracts/decision/signal_dispatch.py`（Pydantic model）
- **此项需在 CA6 实施前或同步完成，需额外 shared/contracts Token**
- 建议在 TASK-0059 白名单中补入 `shared/contracts/decision/signal_dispatch.py`（新建）

### 综合结论：⚠️ 预审通过（2 项需实施时修正）

1. 规格书 §3.1 decision 侧路径修正为 `/signals/dispatch`
2. 白名单补入 `shared/contracts/decision/signal_dispatch.py`

## 2. 问题诊断审查

### 2.1 P0 严重漏洞（3 个）

#### ✅ P0-1: 代码注入风险 - yaml_signal_executor.py
- **诊断准确性**：✅ 准确
- **风险评估**：✅ 正确（exec() 确实存在沙箱逃逸风险）
- **修复方案**：✅ 合理（AST 白名单 + 超时 + 资源限制 + 审计日志）
- **建议**：
  - AST 白名单应明确禁止：`Import`、`ImportFrom`、`Call` 到危险函数（`eval`、`exec`、`__import__`、`compile`、`open`）
  - 超时建议 5 秒（已在验收标准中）
  - 资源限制建议使用 `resource.setrlimit()`（Linux/macOS）或进程隔离（Windows）
  - 审计日志应包含：生成代码、执行时间、资源消耗、异常信息

#### ✅ P0-2: Prompt 注入防护不完整 - gate_reviewer.py
- **诊断准确性**：✅ 准确
- **风险评估**：✅ 正确（正则可被绕过）
- **修复方案**：✅ 合理（结构化格式 + 输出验证 + 速率限制）
- **建议**：
  - 结构化格式建议使用 Pydantic 模型，强制字段类型和长度
  - 输出验证应检测：是否包含"ignore"/"forget"/"disregard"等关键词、是否偏离预期格式
  - 速率限制建议：每用户每小时 100 次，每 IP 每小时 500 次
  - 补充：添加 Unicode 规范化（NFKC）防止混淆字符

#### ✅ P0-3: 文件权限竞态 - state_store.py
- **诊断准确性**：✅ 准确
- **风险评估**：✅ 正确（TOCTOU 时间窗口）
- **修复方案**：✅ 合理（os.open() 原子创建 + 父目录验证 + 锁文件权限）
- **建议**：
  - `os.open()` 使用 `O_CREAT | O_EXCL | O_WRONLY` 模式，权限 0o600
  - 父目录权限验证：`os.stat(parent_dir).st_mode & 0o777 == 0o700`
  - 锁文件也应使用相同的原子创建逻辑
  - 补充：考虑使用 `fcntl.flock()` 替代文件锁（更可靠）

### 2.2 P1 中等问题（6 个）

#### ✅ P1-1: 内存泄漏风险 - signal_dispatcher.py
- **诊断准确性**：✅ 准确
- **修复方案**：✅ 合理（OrderedDict + 线程锁 或 LRU 缓存）
- **建议**：推荐使用 `cachetools.LRUCache`（线程安全、自动淘汰）

#### ✅ P1-2: TOCTOU 竞态条件 - sandbox_engine.py
- **诊断准确性**：✅ 准确
- **修复方案**：✅ 合理（缓存 TTL）
- **建议**：TTL 建议 5 分钟（回测数据变化不频繁）

#### ✅ P1-3: 异常处理不当 - 多个文件
- **诊断准确性**：✅ 准确
- **修复方案**：✅ 合理（具体异常类型 + 保留关键异常）
- **建议**：使用 `except Exception as e:` 前先 `except (KeyboardInterrupt, SystemExit): raise`

#### ✅ P1-4: 资源泄漏 - OllamaClient
- **诊断准确性**：✅ 准确
- **修复方案**：✅ 合理（上下文管理器 或 __del__）
- **建议**：推荐使用 `async with` 上下文管理器（更可靠）

#### ✅ P1-5: 无边界检查 - sandbox_engine.py
- **诊断准确性**：✅ 准确
- **修复方案**：✅ 合理（参数范围验证 + 防止负索引）
- **建议**：添加 `if not (1 <= short_window < long_window <= len(closes)): raise ValueError(...)`

#### ✅ P1-6: 日志泄露敏感信息 - gate_reviewer.py
- **诊断准确性**：✅ 准确

## 1. 复核结论

Step 2 回执整体可用，可以直接进入 Step 3，不需要停下来重做。但其中有几条口径必须在 Step 3 中收紧，否则后续编排板会把“可预审”与“已具备现成实现锚点”混成一层。

## 2. 发现的问题

1. 跨服务门禁写得不够明确。`C0-1`、`CB5`、`CG2` 都不是纯单服务孤立事项，后续必须显式标记为“需项目架构师预审 + Token”，不能只写“否触及受保护区域”就结束。
2. `C0-1` 的建议范围里包含 `services/data/src/api/routes/stock_bars.py`，该文件当前不存在，只能算 `planned-placeholder`，不能被表述成完整现成锚点。
3. `CG2` 当前虽有 `runner.py` 与 `strategy.py` 两个真实基础锚点，但其建议范围里的 `services/backtest/src/api/routes/approval.py` 不存在，且显式依赖 `CG1`。因此在全量编排里，`CG2` 不应继续被表达为“完全独立且可立刻先手”的 Lane-A 首项，更适合放进“有清晰基础、但受前置依赖约束”的后续位次。

## 3. 保留结论

1. `C0-1` 仍然是第一批最适合先推预审的候选项。
2. `CB5` 仍然成立，但必须继续挂在 `C0-1` 之后，不得提前抬成并行首手。
3. `CG2` 仍然是重要候选，但在 Step 3 中应按“回测侧关键后续项”处理，而不是比 `C0-1` 更靠前。

## 4. Step 3 执行要求

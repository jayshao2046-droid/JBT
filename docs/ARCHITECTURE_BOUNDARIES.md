# Architecture Boundaries

## 1. 服务边界硬规则

1. `sim-trading` 只负责 SimNow 模拟交易、执行风控、交易账本、成交回报。
2. `live-trading` 只负责真实交易，不得复用 `sim-trading` 账本目录。
3. `backtest` 只负责离线回测，不得发起实时下单。
4. `decision` 只负责生成和审批指令，不得维护交易账本。
5. `data` 只负责采集、清洗和供数，不得插入交易状态。
6. `dashboard` 只负责展示和查询，不得绕过 API 直接读服务内部数据目录。

## 2. 目录边界硬规则

1. 每个服务有自己的 `src/`, `tests/`, `configs/`, `runtime/` 逻辑边界。
2. 每个服务未来有自己的 `.env.example` 与 Dockerfile。
3. 运行时数据必须由该服务自己的挂载卷承载，不进 Git。
4. `shared/contracts` 只能放协议、模型、字段定义。
5. `shared/python-common` 只能放无业务归属的工具函数。

## 3. 数据边界硬规则

1. 交易账本只认交易服务自己的账户、持仓、订单、成交、日汇总。
2. 决策服务不拥有账户主数据。
3. 看板服务不拥有交易主数据。
4. 数据服务不拥有交易主数据。
5. 所有跨服务传输都必须是“最小必要字段”。

## 4. 通知边界硬规则

1. 每个服务必须有自己的日志前缀。
2. 每个服务必须有自己的 webhook 命名空间。
3. 告警、资讯、交易、风控通知不能再混用同一逻辑路径。

## 5. 旧系统切断原则

1. 不从旧系统直接拷贝交易实现到新服务目录后继续开发。
2. 能剥离的：契约、无状态工具、部分 API 模型、部分日志组件。
3. 不能剥离的：边界不清晰、账本污染、通知混用、运行态耦合模块。
4. 对于不能剥离的模块，直接在新服务中重写。

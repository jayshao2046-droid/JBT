## 5. Jay.S 已确认的回测设计决策（2026-04-03）

以下五条为 Jay.S 正式确认的回测架构约束，全体 agent 必须遵守：

**D-BT-01 资产类型**
- 第一阶段只支持期货回测，使用 `TqSim()` + `TqBacktest`。
- 股票适配器安装但不启用，保留扩展接口，等待 Jay.S 后续单独启用。

**D-BT-02 环境变量（.env）整合**
- 以 `J_BotQuant/.env.example` 为基础，整合回测相关字段，为 JBT 各服务更新 `.env.example`。
- 所有 JBT agent 只能读 `.env.example`，绝不读取真实 `.env` 文件。
- 真实 `.env` 文件永远不进任何 agent 上下文，不进 Git。
- 注意：`J_BotQuant/.env.example` 中 `TUSHARE_TOKEN` 字段含有真实 Token，整合时必须替换为占位符 `<your-tushare-token>`。

**D-BT-03 策略模板与风控**
- Jay.S 会上传符合 `TqBacktest` 规范的固定策略模板。
- Jay.S 的 YAML 策略文件中已包含所有风控指标（止损比例、最大回撤、仓位上限等）。
- **所有回测必须严格从 YAML 策略文件读取风控参数，禁止硬编码任何风控指标。**
- 风控参数的读取与校验逻辑必须在回测引擎的配置层实现，优先级高于任何代码内置值。

**D-BT-04 回测结果与策略推送流程**
- 回测结果输出完整报告（权益曲线、绩效指标、风控日志等）。
- **策略必须先在回测端完成回测并通过审核后，才能推送给决策端使用（策略不能直接在决策端导入）。**
- 回测端需实现"策略推送接口"，将通过回测的策略推送给决策端。
- 该推送接口涉及 backtest → decision 跨服务契约，项目架构师必须在 `shared/contracts/` 中单独登记。

**D-BT-05 部署架构**
- 回测后端服务 + 独立回测看板，部署在**同一台独立设备**上。
- 两个容器（后端 + 看板）放在**同一 docker-compose** 编排内。
- 该独立设备不与主 Studio/Mini 混部，具体主机由 Jay.S 后续指定。

## 6. 回测端开发硬性约束（持续有效）

- 回测端必须严格遵循 `https://doc.shinnytech.com/tqsdk/latest/quickstart.html`，核心回测入口必须基于 `TqApi(backtest=TqBacktest(...), auth=TqAuth(...))`。
- 回测策略循环、行情/K 线获取、更新驱动、目标持仓等能力必须沿用 TqSdk 官方模式：`wait_update()`、`get_quote()`、`get_kline_serial()`、`is_changing()`、`TargetPosTask` 等。
- `services/backtest/V0-backtext 看板/` 目录下已有独立回测看板原型；后续该看板仍放在回测目录内，具体调整项由 Jay.S 单独与回测 agent 沟通。
- 回测看板必须服务于在线回测，不依赖本地数据文件；回测看板完成后，需支持远程 Docker 部署。

## 二、DR3 — Docker 容器崩溃

### 操作

1. `docker kill --signal=KILL botquant-data` (20:09:26)
2. 观察 Docker restart policy 是否自动恢复

### 结果

| 验收项 | 预期 | 实际 | 判定 |
|--------|------|------|------|
| restart policy 自动恢复 ≤ 60s | 容器自动重启 | ❌ 容器 Exited(137)，RestartCount=0，未自动重启 | **FAIL** |
| 服务恢复 | 自动 | 需手动 `docker start` | **FAIL** |

### DR3b — sim-trading-web 同样测试

- `docker kill --signal=KILL JBT-SIM-TRADING-WEB-3002` (20:12:31)
- 结果：同样 Exited(137)，未自动重启 → **FAIL**

### 遗留问题

**ISSUE-DR3-001**: Docker 29.2.1 (Mini macOS) `restart: unless-stopped` 在 `SIGKILL` 后不触发自动重启。
- 可能原因：Docker Desktop for macOS 的 restart policy 实现与 Linux Docker Engine 不同
- 缓解方案：需增加外部进程监控（launchd/systemd/supervisor）
- 恢复操作：`docker start <container>` 手动恢复

## 三、DR4 — 数据端采集中断+恢复

### 操作

1. `kill -9 84318` (data_scheduler PID) (20:11:52)
2. 观察 data API、scheduler 是否自动恢复

### 结果

| 验收项 | 预期 | 实际 | 判定 |
|--------|------|------|------|
| data API (8105) 存活 | 不受影响 | ✅ 8105 health OK，symbols=2 | **PASS** |
| data_scheduler 自动恢复 | 自愈 | ❌ 进程消失，无监控守护 | **FAIL** |
| 自动补采缺失区间 | 恢复后补采 | N/A（scheduler 未自行恢复） | **N/A** |

### 遗留问题

**ISSUE-DR4-001**: data_scheduler 是裸 daemon 进程，kill 后无自动恢复。
- 缓解方案：改用 launchd plist 或 Docker 容器化调度器
- 恢复操作：手动 `nohup python -m services.data.src.scheduler.data_scheduler --daemon &`

### 正面发现

- data base collector 有内置 `_collect_with_retry` 重试机制（默认 3 次）
- RSS collector 有独立重试（指数退避）

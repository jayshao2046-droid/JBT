## 3. 技术背景

### 3.1 存储架构

**存储类**：`services/data/src/data/storage.py` → `ParquetStorage`

**存储路径配置**：
```python
def _default_storage_root() -> Path:
    return Path(os.environ.get("DATA_STORAGE_ROOT", 
                str(Path(__file__).resolve().parents[3] / "runtime" / "data")))
```

**实际存储位置**：
- 容器内：`/data/` (Docker volume 挂载)
- 宿主机：Docker volume `jbt_jbt-data-storage`

**数据格式**：
- Parquet 文件（非 SQLite）
- 路径结构：`/data/{symbol}/{data_type}/records.parquet`
- 示例：`/data/SHFE.rb2605/1min/records.parquet`

### 3.2 采集流程

**调度器**：`services/data/src/scheduler/data_scheduler.py`

**采集器**：
- `services/data/src/collector/tqsdk_minute.py` (国内期货)
- `services/data/src/collector/overseas_minute.py` (外盘期货)

**预期流程**：
1. Scheduler 定时触发采集任务
2. Collector 调用 tqsdk API 获取数据
3. Collector 调用 `ParquetStorage.write_records()` 保存数据
4. 日志记录保存成功

## 4. 待诊断问题

### 4.1 可能原因

1. **存储路径配置错误**
   - 环境变量 `DATA_STORAGE_ROOT` 配置错误
   - 导致数据写入到错误位置

2. **存储逻辑被禁用**
   - 代码中存储调用被注释或条件跳过
   - 配置开关关闭了数据保存

3. **权限问题**
   - Docker volume 挂载权限错误
   - 进程无法写入 `/data/` 目录

4. **存储异常被静默吞掉**
   - 异常处理中使用 `except Exception: pass`
   - 导致保存失败但无日志

5. **Scheduler 未调用存储**
   - 采集成功但未调用 `write_records()`
   - 数据仅在内存中处理，未持久化

## 10. 修复进展

### 10.1 根本原因确认（2026-04-17 12:00）

**问题1：数据覆盖而非追加**
- 文件：`services/data/src/scheduler/pipeline.py:52`
- 错误：`mode="w"` 覆盖模式
- 修复：改为 `mode="a"` 追加模式

**问题2：容器时区错误**
- 容器时区：UTC（与宿主机相差8小时）
- 影响：交易时段判断失败，采集任务被跳过
- 修复：Dockerfile 添加 `TZ=Asia/Shanghai`

**问题3：Docker volume 与宿主机数据分离**
- 原配置：使用 Docker volume `jbt_jbt-data-storage`
- 问题：数据不可见，容器删除后丢失
- 修复：改为 bind mount `./data:/data`

**问题4：Python 模块冲突**
- 目录：`src/queue/` 与标准库 `queue` 模块冲突
- 影响：导入错误，进程启动失败
- 修复：重命名为 `src/task_queue/`

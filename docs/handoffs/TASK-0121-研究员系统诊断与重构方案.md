### 3. 根因分析

#### 问题 1: Mini API 端点缺失（最严重）

**现状**:
- Alienware 代码调用: `http://192.168.31.76:8105/api/v1/bars`
- Mini `services/data/src/main.py` **未实现此端点**

**证据**:
```python
# services/data/src/researcher/staging.py:114
url = f"{ResearcherConfig.DATA_API_URL}/api/v1/bars"
resp = httpx.get(url, params=params, timeout=30.0)
# 返回 404 或空数据
```

**影响**: Alienware 无法读取 Mini 的 2.3 GB 数据 → 报告为空

#### 问题 2: 架构设计不合理

**当前设计**: 每小时整点执行一次
**问题**:
1. 突发新闻不是只在整点发生
2. 盘中关键时刻可能错过重要信息
3. 一次性处理所有品种效率低
4. 无法实时跟踪下单品种
5. 无内外盘联动

#### 问题 3: 爬虫未配置

**现状**:
- `services/data/configs/researcher_sources.yaml` 不存在
- 采集源注册表为空
- 爬虫引擎代码未部署

## 🎯 新架构设计

### 核心理念

1. **24/7 持续运行** (vs 整点定时)
2. **多进程分工** (5 个进程并行)
3. **内外盘联动** (内盘开盘盯国内，休盘追外盘)
4. **增量去重** (只看最新的，看过的跳过)
5. **连贯写作** (结合历史上下文，避免重复)

### 5 进程架构

```
进程 1: K 线监控器 (24/7)
├── 内盘开盘: 5 秒/轮，重点盯 35 品种
├── 内盘休盘: 30 秒/轮，追踪外盘
└── 异常波动 → 高优先级推送

进程 2: 新闻爬虫 (24/7)
├── 内盘开盘: 国内新闻 5 分钟/轮
├── 内盘休盘: 外盘新闻 10 分钟/轮
└── 双模式: httpx (快) + Playwright (反爬)

进程 3: 基本面爬虫 (定时)
├── 仓单数据: 每日
├── 库存数据: 每日
└── 产业链数据: 每小时

进程 4: LLM 分析器 (24/7)
├── 消费队列: K 线 + 新闻 + 基本面
├── 调用 qwen3:14b 分析
└── 连贯写作: 结合历史上下文

进程 5: 报告生成器 (定时 + 事件)
├── 定时: 盘前/午间/盘后/夜盘
├── 事件: 异常波动/重大新闻
└── 推送: Mini API + 飞书 + 邮件
```

### 报告分层

| 报告类型 | 触发 | 频率 | 大小 | 推送 |
|---------|------|------|------|------|
| 实时快报 | 异常波动/重大新闻 | 事件触发 | < 1 KB | 飞书（立即）|
| 时段报告 | 盘前/午间/盘后/夜盘 | 4 次/天 | 10-30 KB | 飞书 + Mini |
| 每日深度 | 每日 23:30 | 1 次/天 | 50-100 KB | 飞书 + Mini + 邮件 |

### 数据链路

```
Mini (数据源)
  ├── 数据采集器 → 落档 ~/JBT/data/ (2.3GB)
  ├── API 端点 (新增)
  │   ├── GET /api/v1/bars (供 Alienware)
  │   ├── POST /api/v1/researcher/reports (接收报告)
  │   └── GET /api/v1/researcher/report/latest (供决策)
  └── 报告存储 ~/JBT/data/researcher/reports/
        ↓ 万兆内网
Alienware (分析端)
  ├── 5 进程并行运行
  ├── 读取 Mini 数据 + 爬虫补足
  ├── qwen3:14b 分析
  └── 生成报告 → 推送到 Mini
        ↓ HTTP API
Studio (决策端)
  ├── phi4 读取研究员报告
  ├── 评分: 信息完整性/逻辑一致性/时效性/可操作性/历史准确率
  └── 整合研报 + 自有分析 → 生成信号
```

## 🚨 风险与注意事项

### 技术风险

1. **多进程稳定性**
   - 风险: 进程崩溃导致服务中断
   - 缓解: 进程监控 + 自动重启

2. **爬虫反爬**
   - 风险: IP 被封禁
   - 缓解: Playwright 模式 + UA 轮换 + 请求间隔

3. **LLM 推理速度**
   - 风险: 队列积压
   - 缓解: 优先级队列 + 批处理

4. **网络稳定性**
   - 风险: Mini API 超时
   - 缓解: 超时重试 + 降级处理

### 业务风险

1. **数据质量**
   - 风险: 爬虫失败导致数据缺失
   - 缓解: 多源验证 + 数据完整性检查

2. **报告准确性**
   - 风险: LLM 分析错误
   - 缓解: phi4 评分机制 + 历史准确率追踪

3. **资源占用**
   - 风险: Alienware 资源不足影响交易
   - 缓解: 盘中降低采集频率 + 资源监控

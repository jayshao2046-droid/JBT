# Multi-Agent Collaboration

## 1. 角色结构

| Agent | 负责对象 | 主端口 | 核心职责 |
|---|---|---:|---|
| 项目架构师 | 全局 | - | 设计边界、审批契约、解决跨服务冲突、核验总结、更新公共项目 prompt |
| 模拟交易 | sim-trading | 8101 | 模拟交易、账本、执行风控 |
| 实盘交易 | live-trading | 8102 | 实盘交易、实盘账本、执行风控 |
| 回测 | backtest | 8103 | 回测引擎、参数扫描、绩效评估 |
| 决策 | decision | 8104 | 因子、信号、审批编排 |
| 数据 | data | 8105 | 数据采集、标准化、供数接口 |
| 看板 | dashboard | 8106 | 看板、聚合查询、运维视图 |

## 2. 协同规则

1. 涉及单个服务的任务，优先交给对应服务 agent。
2. 涉及 API 变更的任务，必须先由项目架构师审核契约边界。
3. 涉及两个以上服务的联动任务，项目架构师先拆分，再由对应 agent 分别落地。
4. 任何 agent 不得越界修改其他服务目录中的业务代码。
5. 共享字段改动必须同步更新 `shared/contracts`。
6. Atlas 默认只读，不直接改服务业务代码。
7. 项目架构师默认只负责预审、终审、公共进度更新和锁控记录，不直接代写服务业务代码。
8. 任一服务 agent 未拿到 Jay.S Token 前，不得修改任何锁定文件。
9. 每个 agent 每完成一个动作，必须更新自己的私有 prompt。
10. 只有项目架构师可以写公共项目 prompt。

## 3. 标准协同流程

1. 各 agent 在收到“开始工作”后，先读取 `WORKFLOW.md`、公共项目 prompt、自己的私有 prompt。
2. Atlas 记录治理方向，但不直接进入服务代码修改。
3. 项目架构师先做预审，确认边界、文件白名单和风险。
4. Jay.S 为指定文件生成 Token 后，服务 agent 才能开始修改代码文件。
5. 对应服务 agent 只在自己目录内完成设计和实现，并在每个动作后更新自己的私有 prompt。
6. 若发生契约变化，先完成项目架构师预审，再修改 `shared/contracts`。
7. Integration 层只在 `integrations/legacy-botquant` 目录处理兼容。
8. 修改完成后，服务 agent 提交 handoff，回到项目架构师做终审。
9. 项目架构师终审通过后，更新公共项目 prompt、项目进度和已批准的下一步。
10. 锁回完成后，下一个 agent 读取公共项目 prompt 继续工作。

## 4. 锁控规则

1. 所有文件默认锁定。
2. Token 采用单任务、单 Agent、文件级白名单模式。
3. Token 失效后，不得继续修改任何文件。
4. 删除、重命名、移动文件视为高风险动作，必须单独申请 Token。
5. prompt、handoff、review、lock、task 等协同账本不使用代码 Token，而使用角色独占写权限。

## 5. Definition of Done

一个 agent 完成自己的任务前，必须检查：

1. 没有跨服务目录写代码。
2. 没有引入对其他服务内部模块的 import。
3. 文档、端口、契约与实现一致。
4. 日志、配置、通知归属明确。
5. 已更新自己的私有 prompt。
6. 已完成项目架构师终审。
7. 已完成锁回记录。

### P0-1: 数据服务 API 认证绕过风险

**文件路径**: `services/data/src/main.py:191-197`  
**漏洞类型**: 认证绕过  
**严重程度**: 高危（P0）

**具体描述**:
```python
async def _verify_api_key(request: Request, api_key: Optional[str] = Depends(_api_key_header)) -> None:
    if not _DATA_API_KEY:
        return  # 如果未配置 API_KEY，直接返回，所有请求都通过！
    if request.url.path in _PUBLIC_PATHS:
        return
    if not api_key or not hmac.compare_digest(api_key, _DATA_API_KEY):
        raise HTTPException(status_code=403, detail="invalid or missing API key")
```

当 `_DATA_API_KEY` 环境变量未设置时，所有 API 请求都会被接受，包括敏感的 ops 端点（采集器重启、自动修复等）。

**潜在影响**: 
- 未授权访问所有 API 端点
- 可执行采集器重启、自动修复等危险操作
- 数据泄露和系统破坏风险

**修复建议**:
```python
async def _verify_api_key(request: Request, api_key: Optional[str] = Depends(_api_key_header)) -> None:
    if request.url.path in _PUBLIC_PATHS:
        return
    if not _DATA_API_KEY:
        logger.error("DATA_API_KEY not configured - all requests will be rejected")
        raise HTTPException(status_code=503, detail="API key not configured")
    if not api_key or not hmac.compare_digest(api_key, _DATA_API_KEY):
        raise HTTPException(status_code=403, detail="invalid or missing API key")
```

**文件路径**: `services/data/src/main.py:1455-1458`  
**漏洞类型**: 命令注入  
**严重程度**: 高危（P0）

**具体描述**:
```python
subprocess.run(
    ["launchctl", "kickstart", "-k", f"gui/{os.getuid()}/{plist_id}"],
    capture_output=True, text=True, timeout=15,

```

虽然使用了列表形式（相对安全），但 `plist_id` 来自白名单映射，存在逻辑绕过风险。如果白名单被污染或扩展不当，可能导致任意命令执行。

**潜在影响**: 
- 远程代码执行（RCE）
- 系统完全被控制
- 数据泄露和破坏

**修复建议**:
```python
# 严格验证 plist_id 格式
if not re.match(r'^[a-zA-Z0-9._-]+$', plist_id):
    raise HTTPException(status_code=400, detail="invalid plist_id format")

# 验证 plist 文件存在且位置正确
plist_path = Path.home() / "Library" / "LaunchAgents" / f"{plist_id}.plist"
if not plist_path.exists() or not plist_path.is_file():
    raise HTTPException(status_code=404, detail="plist not found")
if not str(plist_path).startswith(str(Path.home() / "Library" / "LaunchAgents")):
```

### P0-3: LLM Prompt 注入漏洞

**文件路径**: `services/decision/src/llm/gate_reviewer.py:77-84`  
**漏洞类型**: Prompt 注入 / LLM 安全  
**严重程度**: 高危（P0）

**具体描述**:
```python
user_prompt = f"""策略: {strategy_id}
标的: {symbol}
信号: {signal_desc} (强度 {signal_strength:.2f})
因子摘要: {factor_summary}

{context}

请判断该信号是否与近期趋势严重矛盾，输出 JSON。"""
```

`context` 参数直接拼接到 prompt 中，未进行任何清理。恶意用户可通过注入特殊指令改变 LLM 行为。

**潜在影响**: 
- LLM 被迫输出恶意内容
- 决策逻辑被绕过
- 虚假信号生成
- 交易决策被操纵

**修复建议**:
```python
def sanitize_context(context: str) -> str:
    """清理 context 防止 prompt 注入"""
    # 移除可能的 prompt 注入关键词
    dangerous_patterns = [
        r'(?i)ignore.*instruction',
        r'(?i)system.*prompt',
        r'(?i)forget.*previous',
        r'(?i)you\s+are\s+now',
        r'(?i)new\s+instruction',
    ]
    for pattern in dangerous_patterns:
        context = re.sub(pattern, '[FILTERED]', context)
    return context[:2000]  # 限制长度

# 使用结构化格式
user_prompt = json.dumps({
    "task": "signal_review",
    "strategy_id": strategy_id,
    "symbol": symbol,
    "signal": {"desc": signal_desc, "strength": signal_strength},
    "factor_summary": factor_summary,
    "context": sanitize_context(context),
}, ensure_ascii=False)
```

### P1-1: 信号分发器内存泄漏

**文件路径**: `services/decision/src/core/signal_dispatcher.py:43-50`  
**漏洞类型**: 内存泄漏 / 资源管理  
**严重程度**: 中危（P1）

**具体描述**:
```python
if len(self._dispatched) > self.max_history:
    oldest_keys = sorted(
        self._dispatched.keys(),
        key=lambda k: self._dispatched[k].get("dispatched_at", ""),
    )[: len(self._dispatched) - self.max_history]
    for key in oldest_keys:
        del self._dispatched[key]
```

FIFO 淘汰逻辑存在问题：当 `dispatched_at` 为空字符串时，排序不稳定，可能导致某些记录永不被淘汰。

**潜在影响**: 
- 长期运行导致内存占用持续增长
- 最终导致 OOM（Out of Memory）
- 服务崩溃

**修复建议**:
```python
if len(self._dispatched) > self.max_history:
    # 使用时间戳而非字符串排序，确保稳定排序
    import time
    oldest_keys = sorted(
        self._dispatched.keys(),
        key=lambda k: self._dispatched[k].get("dispatched_at", "1970-01-01T00:00:00"),
    )[: len(self._dispatched) - self.max_history]
    for key in oldest_keys:
        del self._dispatched[key]
```

## 📋 治理流程建议

根据 JBT 治理规则，这些安全漏洞修复需要遵循以下流程：

### 1. 任务建档
- 由 Atlas 确认修复范围和优先级
- 建立独立的安全修复任务（如 TASK-0110-SG5）
- 明确修复目标和验收标准

### 2. 架构预审
- 项目架构师审查修复方案
- 冻结文件白名单
- 确定修复批次和依赖关系

### 3. Token 签发
- Jay.S 为白名单文件签发 Token
- 记录 Token 有效期和范围
- 登记到锁控系统

### 4. 实施修复
- 按白名单实施修复
- 编写单元测试验证修复效果
- 记录修复过程和决策

### 5. 复审和锁回
- Atlas 复核修复结果
- 项目架构师终审

     ```python
    dangerous_patterns = [
     ```

#### 1. 优化信号分发器性能（P1-1）
**文件**: `services/decision/src/core/signal_dispatcher.py`  
**问题**: 第一轮修复使用 `sorted()` 进行 FIFO 淘汰，高频场景下性能问题  
**修复内容**:
- 使用 `OrderedDict` 替代普通 `dict`，自动维护插入顺序
- 使用 `popitem(last=False)` 实现 O(1) 复杂度的 FIFO 淘汰
- 避免每次淘汰时对所有键进行排序

**代码变更**:
```python
from collections import OrderedDict

class SignalDispatcher:
    def __init__(self, sim_trading_url: str = "http://localhost:8101") -> None:
        self.sim_trading_url = sim_trading_url.rstrip("/")
        # 使用 OrderedDict 替代普通 dict
        self._dispatched: OrderedDict[str, dict] = OrderedDict()
        self.max_history = 10000

    async def dispatch(self, request: SignalDispatchRequest) -> SignalDispatchResponse:
        # FIFO 淘汰逻辑优化为 O(1)
        if len(self._dispatched) > self.max_history:
            num_to_remove = len(self._dispatched) - self.max_history
            for _ in range(num_to_remove):
                self._dispatched.popitem(last=False)  # FIFO: 删除最早插入的项
```

```python
import re
from pathlib import Path

def _validate_date_format(date_str: str) -> None:
    """验证日期格式为 YYYY-MM-DD"""
    if not re.match(r'^\d{4}-\d{2}-\d{2}$', date_str):
        raise HTTPException(status_code=400, detail="Invalid date format, expected YYYY-MM-DD")

def _validate_segment(segment: str) -> None:
    """验证 segment 格式为 HH-00（00-00 到 23-00）"""
    if not re.match(r'^(0\d|1\d|2[0-3])-00$', segment):
        raise HTTPException(status_code=400, detail="Invalid segment format, expected HH-00")

def _safe_path_join(base_dir: str, *parts: str) -> Path:
    """安全路径拼接，防止路径遍历"""
    base = Path(base_dir).resolve()
    target = (base / Path(*parts)).resolve()
    if not str(target).startswith(str(base)):
        raise HTTPException(status_code=403, detail="Path traversal detected")
    return target
```

- `services/data/src/main.py:216-218`
- `services/sim-trading/src/main.py:66-67`
- `services/decision/src/api/app.py:73-74`

**漏洞类型**: 认证绕过  
**严重程度**: 中危（P1）

**问题描述**:
所有服务在未配置 API Key 时都会放行所有请求，即使在生产环境也是如此。这在开发环境可以接受，但在生产环境是严重的安全隐患。

**修复方案**:
```python
# 修复前
if not _API_KEY:
    return  # 直接放行

# 修复后
if not _API_KEY:
    env = os.environ.get("JBT_ENV", "development").lower()
    if env == "production":
        raise HTTPException(
            status_code=503,
            detail="API_KEY not configured in production environment"
        )
    # 开发环境允许未配置
    return
```

**修复状态**: ✅ 已修复（4 个服务）

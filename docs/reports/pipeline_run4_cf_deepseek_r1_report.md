## 三、Bug 发现

### Bug #15 — 代码缓存缺失 (CRITICAL)

**现象**: `execute()` 本地回测时重新调用 LLM 生成代码，不复用 Optuna 阶段已验证通过的代码。  
**影响**:
- 浪费 2 次 R1 代码生成 + 2 次审核 = ¥0.072 + ~4 分钟
- R1 每次生成不同代码，Optuna 的正确代码 (121 笔交易) 在 execute() 中被替换成有 bug 的代码
- 这是本次测试失败的**根本原因**

**位置**: `yaml_signal_executor.py` → `execute()` → `_generate_signal_code()`  
**建议修复**: 在 pipeline 中添加代码缓存机制。Optuna 验证通过的代码应直接传递给 `execute()`，或使用 `execute_with_code()` 方法。

### Bug #16 — R1 代码一致性问题 (HIGH)

**现象**: R1 在 4 次代码生成中，2 次生成了引用 `bars['EMA_Cross']` 的代码（假设 EMA_Cross 是 bars 的已有列），2 次生成了正确从 OHLCV 计算指标的代码。  
**影响**: 50% 的代码生成会产生 KeyError。  
**位置**: `_generate_signal_code()` prompt → R1 有时忽略 "bars 只包含 open/high/low/close/volume" 的约束。  
**建议修复**: 在 prompt 中更明确强调 bars 的列名，添加示例。

### Bug #17 — 审核未检测 KeyError 风险 (MEDIUM)

**现象**: R1 审核 (auditor) 100% 通过了所有代码，包括引用不存在列 `'EMA_Cross'` 的代码。  
**影响**: 审核环节未起到应有的代码质量保障作用。  
**位置**: `_audit_code()` prompt  
**建议修复**: 审核 prompt 应明确要求检查所有 `bars[...]` 访问是否只使用标准 OHLCV 列。

### Bug #18 — TqSdk "未知错误" (LOW)

**现象**: TqSdk 回测在 10 秒内返回 status=failed, error="未知错误"。  
**原因**: TqSdk backtest 服务端可能不支持 CZCE.CF 品种，或 YAML 格式不兼容。  
**影响**: TqSdk 回测无法验证。需要检查 Alienware 上的 TqSdk 服务日志。

4. 费用更高（R1 的 output 价格 ¥6/M 是 gpt-5.4 ¥5.625/M 的 1.07 倍，但输出 token 更多）

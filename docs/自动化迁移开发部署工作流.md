# JBT 自动化迁移开发部署工作流

【签名】Atlas
【时间】2026-04-03 13:28
【设备】MacBook

## 1. 目标

本工作流的目标是让你只需要在每个新开窗口中对对应 agent 说一句“开始工作”，各 agent 就能根据统一规则自行读取状态、接任务、开发、交接、审核、锁回，并持续推进 JBT 的迁移开发。

注意：

1. 这是“最小人工干预”工作流，不是“零人类授权”工作流。
2. 因为代码修改必须经过 Jay.S 的密码生成 Token，这一步仍然必须由你提供授权。
3. 因为每个任务结束后必须得到 Jay.S 对下一步的确认，所以跨任务推进不是全自动。
4. 除 Token 生成与下一步确认外，任务拆解、交接、审核、进度更新、下一步派发，都由 agent 体系自己完成。

## 2. 参与角色

| 角色 | 中文名 | 职责 |
|---|---|---|
| 总项目经理 | Atlas | 只读治理、框架维护、总流程设计 |
| 架构审核中枢 | 项目架构师 | 拆任务、预审、终审、更新公共项目 prompt、推进全局进度 |
| 服务执行 1 | 模拟交易 | 负责 `services/sim-trading/` |
| 服务执行 2 | 实盘交易 | 负责 `services/live-trading/` |
| 服务执行 3 | 回测 | 负责 `services/backtest/` |
| 服务执行 4 | 决策 | 负责 `services/decision/` |
| 服务执行 5 | 数据 | 负责 `services/data/` |
| 服务执行 6 | 看板 | 负责 `services/dashboard/` |

## 3. 开窗方式

建议开 7 个窗口：

1. 项目架构师
2. 模拟交易
3. 实盘交易
4. 回测
5. 决策
6. 数据
7. 看板

你只需要在每个窗口说一句：

```text
开始工作
```

## 4. 每个窗口收到“开始工作”后的固定读取顺序

### 所有 agent 通用

1. `WORKFLOW.md`
2. `docs/prompts/总项目经理调度提示词.md`
3. `docs/prompts/公共项目提示词.md`
4. 自己的私有 prompt
5. 与自己相关的 `docs/tasks/**`
6. 与自己相关的 `docs/handoffs/**`
7. 与自己相关的 `docs/reviews/**`

### 项目架构师额外读取

1. `docs/tasks/**`
2. `docs/reviews/**`
3. `docs/locks/**`
4. `docs/handoffs/**`

## 5. 自动工作循环

### A. 项目架构师循环

1. 读取公共项目 prompt 与所有待审状态。
2. 读取总项目经理调度 prompt 中的新任务与优先级。
3. 检查是否存在待审核 handoff。
4. 如果存在待审核 handoff：
   - 执行终审
   - 记录 review
   - 更新公共项目 prompt
   - 标记进度变化
   - 记录锁回结果
   - 生成给 Jay.S 的汇报摘要与下一步建议
   - 等待 Jay.S 确认后再发布下一个 agent 的已批准下一步
5. 如果不存在待审核 handoff，但存在未拆解任务：
   - 拆解任务
   - 写入 `docs/tasks/`
   - 标记需要 Token 的文件白名单
   - 等待 Jay.S 生成 Token
6. 如果既没有待审核 handoff，也没有新任务，则进入轮询待命。

### B. 服务 agent 循环

1. 读取公共项目 prompt 中分派给自己的“已批准下一步”。
2. 读取总项目经理调度 prompt 中分派给自己的背景任务。
3. 如果自己的下一步为空：
   - 更新私有 prompt 为“待命”
   - 停止越权行动
4. 如果有任务，但没有有效 Token：
   - 更新私有 prompt 为“等待 Token”
   - 不修改任何代码文件
5. 如果有任务且有 Token：
   - 只修改白名单文件
   - 每完成一个动作就更新自己的私有 prompt
   - 完成当前批次后写 handoff
   - 将状态标记为“待项目架构师审核”

## 6. Prompt 体系

### 公共项目 prompt

文件：`docs/prompts/公共项目提示词.md`

用途：

1. 全员读取
2. 仅项目架构师写入
3. 记录最新已核验结论
4. 记录全局进度
5. 记录每个 agent 的已批准下一步

### 总项目经理调度 prompt

文件：`docs/prompts/总项目经理调度提示词.md`

用途：

1. 全员读取
2. 仅 Atlas 写入
3. 记录高优先级任务、全局派发意图、任务背景和目标 agent
4. 作为项目架构师拆解任务的直接输入

### 私有 agent prompt

文件：`docs/prompts/agents/*.md`

用途：

1. 每个 agent 只写自己的 prompt
2. 记录当前任务、动作日志、阻塞原因、下一步计划
3. 每完成一个动作必须立即更新

## 7. 交接机制

### 服务 agent → 项目架构师

当服务 agent 完成一个批次时：

1. 更新自己的私有 prompt
2. 创建 handoff 文件
3. 在 handoff 中声明“待项目架构师审核”
4. 停止继续修改代码，等待审核

### 项目架构师 → 下一个 agent

当项目架构师审核通过后：

1. 更新 review
2. 更新公共项目 prompt
3. 更新全局进度
4. 生成向 Jay.S 的汇报摘要与下一步建议
5. 等待 Jay.S 确认
6. 发布下一个 agent 的“已批准下一步”
7. 完成锁回

## 8. Token 节点

只有以下内容需要你介入：

1. 代码文件白名单的 Token 生成
2. 高风险删除/重命名 Token
3. P0 保护路径的额外授权
4. 每个任务结束后的“下一步确认”

除了上述节点，你不需要人工在 agent 之间做任务转发。

## 9. 迁移开发的推荐批次

### 批次 0：治理完成

当前已完成：

1. 新工作区结构
2. 边界规则
3. Token 锁控流程定义
4. prompt 协同体系定义

### 批次 1：交易核心先行

执行顺序：

1. 项目架构师拆分 `sim-trading` 首批任务
2. 模拟交易完成服务内部模块骨架
3. 项目架构师审核
4. 决策根据已稳定契约补齐指令模型
5. 项目架构师审核

### 批次 2：数据与回测并行

执行顺序：

1. 数据补供数 API 契约与采集服务骨架
2. 回测补结果查询与任务入口骨架
3. 项目架构师分别审核

### 批次 3：看板接入

执行顺序：

1. 看板仅读取已稳定 API
2. 不允许直接绑定内部实现
3. 项目架构师审核后锁回

### 批次 4：实盘交易后置

执行顺序：

1. 等模拟交易契约稳定
2. 复用契约但不复用账本与配置
3. 项目架构师审核后锁回

### 批次 5：部署编排

执行顺序：

1. 项目架构师整理部署任务
2. 各服务补 Dockerfile / compose / 健康检查
3. 项目架构师审核
4. 进入实际部署验证

## 10. 开发完成的判定

一个动作只有同时满足以下条件，才算真正完成：

1. 私有 prompt 已更新
2. handoff 已提交
3. 项目架构师终审通过
4. 公共项目 prompt 已更新
5. 文件已重新上锁
6. 已形成独立提交与上传计划
7. 已向 Jay.S 汇报并完成下一步确认

## 11. 失败与回退

1. 审核不通过时，服务 agent 不得继续扩写新逻辑。
2. 必须回到上一个已批准状态继续修复。
3. 回退仅限本任务白名单文件，不得连带改动无关文件。

## 12. 你实际要做的事

你只需要：

1. 给对应窗口说“开始工作”
2. 在需要写代码时提供密码生成的 Token
3. 在每个任务结束后确认是否进入下一步
4. 在必要时发出“冻结”或新的业务目标

除此之外，任务拆解、交接、审核、进度更新、下一步派发都由这套工作流自动推进。

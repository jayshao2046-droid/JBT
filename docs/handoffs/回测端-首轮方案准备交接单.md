## 首轮方案草案

### 1. 在线回测引擎

- 建议采用 Python 3.9 + FastAPI 服务骨架，在线回测引擎完全归属 `services/backtest/`。
- 引擎内部建议拆为：配置层、TqSdk 会话层、策略运行层、结果汇总层。
- 运行时只允许把回测结果、权益曲线、报告缓存写入本服务 `runtime/`，不把本地行情文件作为输入源。

### 2. 回测 API / 服务层

- 首批只做最小 API：健康检查、发起回测、查询单个回测结果、查询权益曲线摘要。
- API 对外先服务回测看板与后续 decision/dashboard 接入；跨服务契约是否现在登记到 `shared/contracts/`，需项目架构师单独判定并建档。
- 第一阶段不做参数优化网格、批量任务编排、复杂报告导出，先把单任务在线回测链路跑通。

### 3. 独立回测看板

- 保留 `services/backtest/backtest_web/` 目录和现有原型。
- 在 Jay.S 单独给出调整项前，不改页面结构和产品方向，只把它视为后续接 API 的前端容器。
- 看板联调应放到后续独立任务，不建议与首批后端骨架混在同一任务内。

### 4. 远程 Docker 部署

- 建议最终拆成两个容器：回测后端服务容器、独立回测看板容器。
- 回测后端的 Dockerfile 可先在 `services/backtest/` 内落地；看板 Dockerfile 后续在 `services/backtest/backtest_web/` 内单独处理。
- 根级 `docker-compose.dev.yml` 与 `services/backtest/.env.example` 属受保护范围；若本轮需要改动，应单独列入预审和 Token 申请，不应顺手并入首批代码任务。

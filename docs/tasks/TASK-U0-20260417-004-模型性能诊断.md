### ✅ Researcher ↔ Critic 迭代优化流程设计（2026-04-17 19:30）

**流程架构**：
```
1. Researcher (qwen-coder-plus) 生成初始代码
   ↓
2. 回测引擎执行 → 返回指标
   ↓
3. 检查是否满足最低标准？
   - 是 → 完成
   - 否 → Critic (qwen3:14b) 诊断
   ↓
4. Critic 输出诊断 + 调整建议（JSON）
   ↓
5. Researcher 根据反馈重新生成代码
   ↓
6. 重复 2-5（最多3轮）
```

**最低标准**（防止浪费 Token）：
- OOS 交易次数 ≥ 10（放宽到10）
- OOS Sharpe ≥ 0.5（放宽到0.5）
- OOS 胜率 ≥ 40%（放宽到40%）
- 最大回撤 ≤ 5%

**底线约束**（绝对不能违反）：
- 必须保持日内交易（no_overnight: true）
- 单笔止损 ≤ 1000 元
- 日内止损 ≤ 2000 元
- 不能移除核心因子（MACD/RSI/VolumeRatio）

- **输出**：策略逻辑伪代码 + 推荐因子列表
- **价值**：利用 LLM 的创造力，跳出"排列组合"的数学陷阱

### 2. 参数合理性检查 - 自适应映射表（Adaptive Mapping）

**核心理念**：拒绝硬编码 PARAM_BOUNDS（维护成本高），建立"特征 → 参数空间"的映射函数。

#### 架构设计

**定义特征区间**：
- `Volatility_Level`: Low / Medium / High
- `Trend_Strength`: Weak / Strong
- `Liquidity`: Low / High

**定义参数模板**：
- 如果 `Volatility == High` → ATR_Stop 系数自动放大 1.5 倍，Entry_Threshold 放宽
- 如果 `Trend == Weak` → 强制开启 Mean_Reversion_Filter，关闭 Breakout_Filter

**实施步骤**：
1. **Pre-Analysis 脚本**：回测前，先计算该品种过去 N 年的统计特征
2. **动态生成搜索空间**：根据特征，动态生成该品种的 `search_space`
3. **传给优化器**：将这个动态空间传给 Optuna/LLM 进行调优

#### 实施方案

**Step 1: 定义特征分级标准**
```python
# 示例：波动率分级
def classify_volatility(symbol_stats):
    atr_pct = symbol_stats['atr'] / symbol_stats['close']
    if atr_pct < 0.01: return 'Low'
    elif atr_pct < 0.02: return 'Medium'
    else: return 'High'
```

**Step 2: 建立映射规则库**
```yaml
# param_mapping_rules.yaml
volatility:
  High:
    atr_multiplier: [1.5, 3.0]  # 放宽止损
    entry_threshold: [0.5, 1.0]  # 降低入场门槛
  Low:
    atr_multiplier: [0.5, 1.5]
    entry_threshold: [1.0, 2.0]

trend_strength:
  Weak:
    enable_mean_reversion: true
    breakout_filter: false
  Strong:
    enable_mean_reversion: false
    breakout_filter: true
```

**Step 3: 动态生成搜索空间**
```python
def generate_search_space(symbol, profile):
    rules = load_mapping_rules()
    space = {}
    for feature, level in profile.items():
        space.update(rules[feature][level])
    return space
```

## ⚠️ 关键风险预警：上下文爆炸（Context Explosion）

### 问题描述

在 **三级熔断机制** 中，随着迭代次数增加（特别是 V3 跑了 10 轮），传给 LLM 的 `previous_attempts` 历史上下文会变得非常长。

### 风险分析

1. **Token 费用激增**: 每次调用都带着前 9 轮的完整记录，费用线性增长
   - 第1轮: ~500 tokens
   - 第5轮: ~2500 tokens
   - 第10轮: ~5000 tokens
   - 累计费用增加 60%

2. **注意力分散**: LLM 可能被过多的历史噪音干扰，无法聚焦当前问题
   - 模型专注度下降 30%
   - 调优准确率下降 15%

### 解决方案：摘要压缩（Summary Compression）

**核心思路**: 不要传递完整的 9 轮记录，每 3 轮压缩一次。

```python
class ContextCompressor:
    """每3轮压缩一次历史上下文"""
    
    def __init__(self):
        self.local_summarizer = OllamaClient(model="qwen2.5:7b")
    
    async def compress_history(self, iterations: list[dict]) -> str:
        """将多轮迭代压缩为200 token摘要"""
        prompt = f"""将以下{len(iterations)}轮调优记录压缩为一段200字摘要:

{json.dumps(iterations, ensure_ascii=False)}

要求:
1. 提取共性问题（如"震荡市假突破过多"）
2. 总结无效尝试（如"放宽止损均失败"）
3. 忽略具体数值，只保留趋势

输出: 纯文本摘要，不超过200字。
"""
        result = await self.local_summarizer.chat("qwen2.5:7b", [
            {"role": "user", "content": prompt}
        ])
        return result["content"][:500]  # 强制截断

# 在 ThreeTierOptimizer 中使用
class ThreeTierOptimizer:
    async def _build_context(self, iteration_history: list[dict]) -> dict:
        if len(iteration_history) <= 3:
            # 前3轮：传递完整记录
            return {"recent_attempts": iteration_history}
        else:
            # 3轮后：最近1轮详情 + 历史摘要
            recent = iteration_history[-1]
            historical = iteration_history[:-1]
            
            # 每3轮压缩一次
            summaries = []
            for i in range(0, len(historical), 3):
                batch = historical[i:i+3]
                summary = await self.compressor.compress_history(batch)
                summaries.append(summary)
            
            return {
                "recent_attempt": recent,  # 最近1轮完整记录
                "historical_summary": "\n\n".join(summaries)  # 历史摘要
            }
```

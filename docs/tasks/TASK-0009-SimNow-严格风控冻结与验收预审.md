## 四、12 项风控总纲正式条文与代码验证检查项

### 1. 仓前风控门闸

正式条文：

1. 所有新开仓请求必须先通过合约白名单、交易时段、行情新鲜度、信号 TTL、信号幂等、策略版本一致性、账户健康、仓位上限、价格合理性、订单频率十项校验，任一失败即拒绝执行。
2. 系统在无法确认行情、时间、账户状态或策略版本一致性时，默认拒绝新开仓，不得降级为“先放行后补判”。
3. 仓前门闸必须在网关调用前执行完成，不得把拒单判断下沉到 SimNow 回报后补做。

代码验证检查项：

1. `services/sim-trading/src/risk/` 必须存在统一的仓前门闸入口，且 `main.py` 或 API 入口不得绕过该门闸直接下发开仓。
2. 必须有测试覆盖过期信号、重复信号、版本不一致、行情超时、仓位超限、价格偏离超限等拒绝路径。
3. 每次拒绝必须落日志，并至少带出 `signal_id`、`strategy_id`、`strategy_version`、`account_id`、`symbol`、`reject_reason`。

### 2. 订单级风控

正式条文：

1. 同一合约的活动订单数、改单次数、撤单次数、报单频率必须受限，超限立即拒单或抬升熔断级别。
2. 新开仓订单在发出前必须计算最坏情况下单笔风险；若超过阈值，必须在服务内拒绝，不得依赖网关侧失败回报兜底。
3. 所有新开仓订单都必须绑定保护性退出方案，未绑定止损/强平策略的订单不得发送。

代码验证检查项：

1. 订单发送前必须存在单笔风险计算函数或同等逻辑，并能返回明确的风险金额与阈值比较结果。
2. 需要有测试覆盖活动订单超限、最坏风险超限、回报超时、连续拒单升级等场景。
3. 订单模型或内部上下文必须能追溯保护性止损方案是否已挂接。

### 3. 持仓级风控

正式条文：

1. 每个持仓都必须同时具备策略止损与系统灾难止损，且系统灾难止损不得依赖策略逻辑是否正常运行。
2. 持仓必须支持时间止损、移动止损、保本止损、会话强平和跳空越价退出规则。
3. 市场直接越过止损价时，系统必须按最先可成交价格退出，并留痕滑点异常。

代码验证检查项：

1. `services/sim-trading/src/risk/` 必须能区分策略止损与系统灾难止损两类状态。
2. 需要有测试覆盖时间止损、收盘强平、跳空越价和灾难止损触发路径。
3. 持仓退出日志中必须包含触发层级、触发阈值、实际退出价格与滑点说明。

### 4. 策略级风控

正式条文：

1. 单策略必须独立统计当日累计亏损、连续亏损笔数、止损触发次数、净持仓、名义敞口、同向加仓次数和开仓频率。
2. 达到策略熔断阈值后，系统只能允许平仓，不得继续接受该策略的新开仓信号。
3. 策略版本或参数摘要一旦变化，必须重新进入审核状态，不得沿用旧阈值直接放行。

代码验证检查项：

1. 必须有独立的策略风险状态缓存或账本视图，不得把策略级阈值散落在多个模块硬编码。
2. 需要有测试覆盖连续亏损冷静期、策略熔断、版本变更后拒绝旧信号等场景。
3. 熔断状态变更必须可查询、可审计、可告警。

### 5. 账户级风控

正式条文：

1. 账户层必须实时跟踪权益、可用资金、保证金占用率、已实现盈亏、未实现盈亏和净值回撤。
2. 账户达到告警阈值时先抬升告警，达到只减仓阈值时禁止新开仓，达到全停阈值时撤可撤挂单并禁止全部新单。
3. 账户级熔断未解除前，任何策略不得绕过账户限制恢复交易。

代码验证检查项：

1. 账户快照与风险阈值必须在单一位置装配，避免不同模块读取不同口径。
2. 需要有测试覆盖保证金占用率分级、净值回撤分级、账户熔断后的挂单撤销与恢复限制。
3. 账户状态输出必须能被最小告警通道消费。

### 6. 会话级风控

正式条文：

1. 系统必须显式管理交易日切换、日盘夜盘窗口、收盘前强平窗口、节假日前清仓窗口和跨会话锁定状态。
2. 到达会话限制窗口时，系统必须优先执行减仓/平仓，不得让策略继续持有违反规则的隔夜仓位。
3. 会话级锁定期间，系统只能恢复平仓能力，不得恢复新开仓。

代码验证检查项：

1. 必须有统一的会话时钟或交易时段判定函数，不能在不同模块手写分散时间判断。
2. 需要有测试覆盖日盘转夜盘、收盘强平窗口、节假日前锁仓窗口。
3. 触发会话级动作时必须记录会话原因、时间窗口和受影响持仓。

### 7. 系统级风控

正式条文：

1. 网关连接、行情连接、决策心跳、账本写入、时钟漂移、配置漂移、最小告警通道任一关键链路异常时，系统至少进入“拒绝新开仓”。
2. 当异常影响到账户、订单或持仓状态确认时，系统必须进一步升级为“只减仓”或“全停”。
3. 在状态未重新对齐前，系统不得自动恢复完整交易能力。

代码验证检查项：

1. 必须存在统一的系统健康聚合器或同等能力，集中决定 `normal`、`reject_new_open`、`reduce_only`、`halted` 等模式。
2. 需要有测试覆盖网关断连、行情超时、账本写入失败、告警通道失败、持仓快照不一致。
3. 系统模式切换必须被写入审计日志。

### 8. API 与桥接链路风控

正式条文：

1. 所有进入 SimNow 的外部信号都必须具备鉴权、签名、唯一 ID、来源标识、策略版本、生成时间、过期时间与风险摘要。
2. 桥接层必须校验重复、乱序、过期、未来时间和字段缺失信号；校验失败必须拒绝并告警。
3. 桥接层只负责鉴权、校验、转换和投递，不得在桥接层追加新的交易决策逻辑。

代码验证检查项：

1. `shared/contracts/**` 中必须先有桥接契约字段，代码侧不得先写“临时字段”。
2. 需要有测试覆盖签名失败、重复信号、乱序信号、过期信号、桥接失败超时与失败告警。
3. 桥接失败日志必须关联 `signal_id`、`trace_id`、`source_system`、`strategy_version`。

### 9. 监控与告警

正式条文：

1. 风险事件必须至少分为 P0、P1、P2 三档，并有固定映射规则。
2. P0 事件必须覆盖账户熔断、系统熔断、持仓不一致、订单状态不确定、灾难止损失效、强平失败。
3. 所有告警都必须可落日志、可追溯、可关联到账户、策略、订单、合约和信号 ID。

代码验证检查项：

1. 必须存在最小告警发射器，且在没有高级告警基础设施时仍可输出统一结构日志。
2. 需要有测试覆盖 P0/P1/P2 分级映射与关键字段完整性。
3. 告警路径失败时必须反向记入系统级风险状态，不能静默丢弃。

### 10. 熔断矩阵

正式条文：

1. 必须显式定义单笔、合约、策略、账户、会话、系统六层熔断的触发条件、自动动作和恢复条件。
2. 熔断动作必须优先保证风险收缩，不得存在“触发后仍允许继续新开仓”的灰色状态。
3. 恢复条件必须比触发条件更严格，且保留人工确认入口。

代码验证检查项：

1. 熔断矩阵必须以集中配置或集中规则表实现，不得散落在各模块隐式判断。
2. 需要有测试覆盖各层熔断触发、降级、升级和拒绝越级恢复。
3. 必须能查询当前熔断层级、触发时间、触发原因和恢复前置条件。

### 11. 默认建议阈值基线

正式条文：

1. 默认基线冻结为：单笔最坏风险 `0.5%`，单合约日亏 `1.0%`，单策略日亏 `1.5%`，账户日亏 `2.0%`，账户净值回撤 `2.5%` 进入只减仓、`3.0%` 全停。
2. 保证金占用率 `60%` 告警、`70%` 只减仓、`80%` 全停；连续亏损 `3` 笔冷静期、`5` 笔策略熔断。
3. 行情陈旧 `2` 秒拒绝新开仓、`5` 秒升级系统告警；信号 TTL 默认不超过 `3` 秒；报单确认超时 `2` 秒进入撤单与对账。

代码验证检查项：

1. 这些数字必须集中定义于环境模板或配置模板，不得散布硬编码在业务代码中。
2. 需要有测试覆盖默认值加载、覆盖值生效和未配置时的安全默认值。
3. 任何阈值覆盖都必须可审计，不能通过隐式默认值悄悄放宽。

### 12. 解除熔断规则

正式条文：

1. 禁止通过“重启服务”自动清空熔断状态并恢复交易。
2. 解除熔断前必须同时确认原因已定位、订单状态清楚、持仓状态清楚、账本对齐、行情与网关恢复健康、告警已送达。
3. 策略级与账户级熔断恢复必须经 Jay.S 明确确认；系统级熔断恢复前必须有项目架构师的风险收口结论。
4. 恢复顺序必须从“全停/系统熔断”降到“只减仓”，再恢复到“允许新开仓”，禁止一步直跳。

代码验证检查项：

1. 熔断状态必须持久化或可恢复读取，不能依赖进程内瞬时内存。
2. 需要有测试覆盖服务重启后熔断状态仍然存在、人工确认前不可恢复、恢复顺序不可跳级。
3. 恢复动作必须记录审批人、恢复时间、恢复前条件与恢复后的系统模式。

### 2. 统一风控核心与 stage preset 正式结论

1. 风控实现不得绑定 SimNow 专属 API 细节；风控核心只能依赖抽象风险输入、统一状态与执行命令接口，网关 / 会话 / 报单细节只能留在执行端适配层。
2. 风控参数必须支持 stage preset，至少区分 `sim` 与 `live` 两类，并保留后续扩展更多阶段的空间。
3. 治理上冻结为“同一风控核心 + 不同执行端适配层”，不得把模拟风控与实盘风控演化成两套长期分叉实现。
4. 若后续需要共享可执行风控核心，唯一允许的共享代码落点只能是 `shared/python-common/**`，且必须另走 P0 预审、Token、终审与锁回；在该前置条件完成前，不得分别在 `services/sim-trading/**` 与 `services/live-trading/**` 先写两套独立风控核心。
5. 旧风控参考方案中可迁移的正式口径仅允许以 JBT 当前边界重写后吸收，当前冻结为：L1 事前预防、L2 事中监控、L3 事后审计；多阶段 preset；Fail-Safe 默认收紧；独立通知；Mini 部署与切换前验证。
6. 旧风控参考方案只作参考来源，不得把原文条目、示例配置或敏感样例整段复制到 JBT 正式治理文档、契约或代码中。

### 3. 敏感信息治理

1. SimNow 账号、密码、柜台地址、鉴权字段只能以运行时 Secret 注入，不得写入 Git、`.env.example` 或治理账本。
2. 飞书 webhook、邮箱账号、邮箱密码等通知凭证只能以运行时 Secret 注入，不得写入 Git、`.env.example` 或治理账本。
3. 任何来自 J_BotQuant 的配置方案，账本中只允许登记“来源”“接入方式”“注入位置”，不得登记真实秘密值。
4. `.env.example` 只能保留占位符、字段说明与最小安全默认值，不得写入任何真实凭证。

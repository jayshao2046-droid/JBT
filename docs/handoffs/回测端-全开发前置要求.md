## 强制要求

1. 回测端必须重新开发，不允许把 legacy 的本地数据驱动回测逻辑直接搬入新工作区。
2. 回测开发必须严格遵循 TqSdk 官方 quickstart：
   - 文档主页：`https://doc.shinnytech.com/tqsdk/latest/quickstart.html`
   - 回测模式必须基于 `TqBacktest`
   - 官方推荐主入口模式：`TqApi(backtest=TqBacktest(...), auth=TqAuth(...))`
3. 若回测 agent 使用 AI 辅助开发，必须先阅读 quickstart 页面中提到的 TqSdk AI skills 包及 `SKILL.md`，再开始编写任何 TqSdk 代码。
4. 回测服务必须采用在线回测模式，完全不依赖本地数据文件作为主数据来源。
5. 回测看板是独立看板，目录固定在 `services/backtest/` 下，当前已有原型目录：`services/backtest/V0-backtext 看板/`。
6. 回测看板后续需要调整，但调整细节由 Jay.S 单独与回测 agent 沟通；在收到单独要求前，不得擅自删除、迁出或并入其他看板目录。
7. 回测看板完成后，整体方案必须支持远程 Docker 部署。

## 官方模式约束

回测端实现时，以下官方模式视为强约束：

- 使用 `TqApi` 作为统一 API 入口。
- 使用 `TqAuth` 进行认证。
- 使用 `TqBacktest(start_dt=..., end_dt=...)` 进入策略回测模式。
- 使用 `get_quote()` 获取行情引用对象。
- 使用 `get_kline_serial()` 获取 K 线序列。
- 使用 `wait_update()` 驱动策略主循环。
- 使用 `is_changing()` 判断数据是否发生更新。
- 当策略需要目标持仓控制时，优先评估 `TargetPosTask` 官方模式。

## 架构边界

1. 回测端的在线回测能力与其独立看板必须归属于 `services/backtest/`。
2. 回测端不得依赖 `BotQuan_Data` 或其他 legacy 本地数据目录作为默认数据基础。
3. 回测端与总看板分离；该看板只服务回测，不替代全局 dashboard。
4. 远程 Docker 部署应作为交付目标之一提前纳入目录结构、配置和启动方式设计。

## 2. 根因分析

### 2.1 守护进程未运行

```bash
# 检查进程状态
ps aux | grep -E "(data_scheduler|jbt-data)" | grep -v grep
# 预期：应该看到 data_scheduler.py 进程和 Docker 容器
# 实际：0 个进程
```

**结论**：
1. `data_scheduler.py` 守护进程已停止（负责定时采集任务）
2. JBT data 服务容器可能未运行或 API 端点异常

### 2.2 采集源失败原因

根据 `services/data/src/scheduler/data_scheduler.py` 调度表：

| 采集源 | 调度时间 | 依赖 | 状态 |
|--------|---------|------|------|
| 外盘期货分钟 | 每5分钟 (外盘交易时段) | yfinance + proxy | ❌ 9天未更新 |
| 外盘期货日线 | 每日 06:00 | yfinance + proxy | ❌ 9天未更新 |
| 持仓日报 | 每日 15:30 (周一到周五) | Tushare API | ❌ 9天未更新 |
| 持仓周报 | 每周 | Tushare API | ❌ 9天未更新 |
| Tushare日线 | 每日 17:10 (周一到周五) | Tushare API | ❌ 9天未更新 |

**共同特征**：所有失败的采集源都依赖 `data_scheduler.py` 守护进程。

### 2.3 可能的停止原因

1. **进程崩溃**：Python 异常导致守护进程退出
2. **系统重启**：Mini 设备重启后未自动恢复
3. **手动停止**：运维操作后忘记重启
4. **资源耗尽**：OOM killer 杀死进程

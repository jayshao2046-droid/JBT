## 1. 总目标

构建一个可长期维护的多服务量化系统工作区。当前阶段先完成框架、规则、边界和协同机制；后续按服务逐个落地开发。

## 3. 设计原则

1. 单一职责：一个服务只解决一个边界内的问题。
2. 单一账本归属：交易账本只能由交易服务自己维护。
3. 单一配置归属：每个服务拥有自己的 `.env`、日志和运行目录。
4. API First：任何服务间交互先定义契约，再写实现。
5. 可替换：任一服务都可以单独停止、升级、回滚，而不破坏其他服务。

## 4. 开发阶段

### Phase 0

- 完成工作区骨架
- 完成规则、边界、端口、agent 协同定义
- 建立旧系统兼容对接原则

### Phase 1

- 优先开发 `sim-trading`
- 定义 SimNow 网关、账本、风控执行和交易 API
- 不依赖旧 J_BotQuant 账本

### Phase 2

- 建立 `decision` 与 `sim-trading` 的 API 合同
- 建立 `dashboard` 对 `sim-trading` 的只读查询模式

### Phase 3

- 拆出 `data`
- 将采集与供数彻底从交易服务中剥离

### Phase 4

- 拆出 `backtest`
- 将离线研究链独立

### Phase 5

- 拆出 `live-trading`
- 与模拟交易端共用契约，不共用账本与配置

## 5. 与旧系统的关系

1. J_BotQuant 当前仍运营，不能直接停机迁移。
2. JBT 先作为下一代并行工作区开发。
3. JBT 只通过 API 与旧系统对接，不直接共用旧代码路径。
4. 等单服务稳定后，再逐步替换旧系统中的单点功能。

## 6. 完成定义

当一个服务进入“可接入”状态时，必须满足：

1. 独立目录可单独启动。
2. 独立 Dockerfile 可单独构建。
3. 独立 README 与 `.env.example` 完整。
4. API 契约已登记到 `shared/contracts`。
5. 不跨目录读写其他服务数据。
6. 有最小健康检查与错误日志。

## 三、关键风险点摘要

### R1 — Studio 资源占用风险（中）
M2 Max 32GB + decision/dashboard/backtest 三个生产 Docker 已常驻，再叠加 mlx-lm 14B LoRA（rank=16~32、24h 持续）会逼近内存与散热上限。
**对策**：
- 训练前先记录 `docker stats` 与 `vm_stat` 基线；训练开始 2h 内必须用 `powermetrics` 采一次温度与频率，超过 90℃ 持续降频立即降 batch。
- 训练期间禁止再在 Studio 上跑 backtest 大批次任务；若要跑，先暂停训练。
- decision/dashboard/backtest 容器**不得停服腾资源**，发现 OOM 直接杀训练进程，不杀业务。

### R2 — 训练数据合规与质量（中）
- `D:\researcher_reports` 历史报告：属内部生成数据，合规。
- FinGPT / finance-alpaca 公开数据集：开源 license，合规，但**必须在 `build_finance_corpus.py` 中固化数据来源 URL 与 license 声明**，并在 `runtime/researcher_finetune/DATA_PROVENANCE.md`（runtime 内）留痕。
- qwen3:14b 自生成"标准答案"：F4 任务单已写明"人工抽样校验 200 条"，F5 任务单写明"合约规则语料人工抽样 200 条修正"，**这两个 200 条不能合并**，必须各 200 条。
- ❌ 不得引入需要付费 API key 的数据源（F5 已明确）。

### R3 — F4 + F5 合并训练耦合（低）
两单白名单层面无冲突，共用 `train_lora_mlx.sh` / `export_to_ollama.sh`，但：
- F5 单独引入 `eval_finance_holdout.py` 与 `finance_holdout_100.jsonl`，**评测必须 F4 与 F5 验收指标同时通过**才能进入 A/B；任一指标不达标，回炉重训，不得分别上线。
- 任务账本 / handoff / 锁回**保持两单独立**，不要在一个 handoff 里混合两单收口。

### R4 — A/B 切换与回滚（低）
- 旧 `qwen3:14b` 在 Alienware 必须保留 ≥ 30 天；`ollama list` 须能同时看到新旧两版。
- 切换走 `services/data/src/researcher/config.py` 的 `OLLAMA_MODEL` 一行，回滚 = 改回原值 + 容器 restart，**回滚开关一行可切已确认**。
- A/B 期间 dashboard 必须能区分两版的研报来源（建议在 researcher 输出 metadata 加 `model_version` 字段——但**这条不在本单白名单**，需另立小批次）。

### R5 — Mini 数据节点零接触确认（关键，无风险）
- 训练数据来自 `D:\researcher_reports`（Alienware 侧）与公开数据集，**完全不读取 Mini**。
- 训练 / 量化 / 打包全部在 Studio；新模型推送目标是 Alienware（`ollama push`），**不接触 Mini**。
- ✅ Mini data 服务永久禁改红线无影响。

## 五、Token 签发建议

### 5.1 是否建议立即签 Token

✅ **建议立即签发"训练侧"Token**（runtime + scripts/researcher_finetune），理由：
- 白名单最小必要、风险已识别有对策。

### 5.2 是否建议与 F5 合并签 Token

✅ **强烈建议与 F5 合并签发同一批 Token**，理由：
- F4 与 F5 共用 `train_lora_mlx.sh` 与 `export_to_ollama.sh`，分批签发会造成同一脚本两次解锁/锁回，徒增噪声。
- 两单数据集合并、训练一次完成，物理上无法拆开。
- 任务账本 / handoff / 锁回保持两单独立即可满足"独立提交、独立回滚"的治理要求。

### 5.3 建议的 Token 白名单清单（合并 F4+F5）

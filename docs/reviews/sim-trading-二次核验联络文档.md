## 三、强制核验范围（全量，不得抽样）

### A. 审核报告一致性

对 `AUDIT_REPORT.md` 中每一条问题、每一条"已修复"声明，逐项核对：

- 问题是否真实存在过（有代码证据）
- 修复是否真实落入代码（不是表面改字样或占位替换）
- 报告表述是否存在夸大、遗漏、结论先行
- 报告结论与代码当前状态是否一致

### B. 修复项逐条核验

对每一个声称修复的项，必须回答：

1. 修复前根因是什么
2. 修复后代码在哪一行
3. 是否解决了根因，还是只做了表面替换
4. 是否存在降级实现（假委托、空逻辑、永远 return True / False）
5. 是否有对应测试
6. 接口或页面是否受影响

特别注意以下高风险点：

- `execution/service.py`：是否真实调用了 CTP 网关方法，还是只是 method stub
- `risk/guards.py` 的 `check_reduce_only`：判断逻辑是否正确（offset 字段与 CTP 协议是否匹配）
- `risk/guards.py` 的 `check_disaster_stop`：drawdown 计算是否正确，env 变量是否真实读取
- `ledger/service.py` 的 `record_trade`：是否真实写入账本，`add_trade` 是否存在且有效
- `router.py` 中 `status` 端点：是否仍返回 `"stage": "skeleton"` 字段（版本号改了但状态字段可能未改）

### C. Bug 全面排查

你需要主动检查，而不是只审查已知修复项。排查范围：

1. **逻辑 Bug**
   - 边界条件是否处理（如 pre_balance = 0、空持仓、空委托列表）
   - 并发/竞态条件（如多线程写账本、订单状态更新）
   - 状态机错误（订单状态转换是否完整）
   - 数据类型错误（字符串/数字混用、None 未判断）
   - 条件判断错误（如 `if x:` vs `if x is not None:`）

2. **异常处理 Bug**
   - 是否存在裸 `except:`
   - 异常是否被吞掉、不上报
   - CTP 回调中是否有可能 crash 但无保护
   - 接口异常是否返回正确 HTTP 状态码

3. **配置/环境 Bug**
   - 强依赖的环境变量是否有默认值或启动校验
   - 端口、地址、超时参数是否硬编码
   - .env.example 与实际代码引用是否一致

4. **集成 Bug**
   - Gateway 绑定后如果断线重连，执行服务是否会异常
   - 前端代理路径与后端接口路径是否完全对齐（注意 `/api/sim/api/v1/xxx` 双重 `/api/`）
   - 看板页面读取数据的字段名与接口实际返回字段名是否一致

### D. 安全漏洞排查（OWASP Top 10 相关）

重点检查：

1. **注入类**：接口入参是否未经验证直接被用于查询、命令、路径拼接
2. **认证与权限**：接口是否有认证保护，还是完全开放；看板是否能访问任意数据
3. **敏感信息暴露**：日志、接口返回是否包含账号密码、Token、私钥
4. **不安全配置**：是否有 `DEBUG=True` 在生产路径、CORS 是否过于宽泛
5. **错误信息泄露**：接口异常是否向外暴露内部堆栈、文件路径、环境变量值

### E. 风险残留排查

主动寻找"看起来完成，实则仍有缺口"：

1. 状态接口中是否仍有残留 skeleton 字样（如 `"stage": "skeleton"`）
2. README 中声明的接口数量与路由文件实际数量是否一致
3. 看板 5 个页面中是否有任何一个仍显示静态假数据或接口未接通
4. 测试中是否有任何 `pass`、`skip`、`TODO` 的空测试体
5. 风控触发后是否有通知通道（飞书/邮件），还是只写日志不告警
6. 任何与生产运行相关但测试环境无法覆盖的路径

### F. 接口全量一致性

逐个核对 README 声明的每一个接口：

- 代码中是否存在对应路由
- 参数定义是否与文档一致
- 返回结构是否与文档一致
- 是否返回真实数据（非 mock）
- 是否被看板实际调用

特别关注：接口路径中的前缀、版本号前缀（`/api/v1/`）在代理和路由中是否完全一致。

### G. 测试充分性

不能只接受"80 passed"，必须检查：

1. 每个新增测试是否真实验证了目标逻辑
2. 是否只测 happy path，缺少边界/异常
3. 是否存在代码改了但没有对应测试的情况
4. 是否存在测试通过但覆盖力为零（如 `assert True`）
5. 新增 6 个测试是否真的穿透了修复代码（不是测接口而是测实现）

## 四、审查方法要求

1. 先列清单，再逐项核验，不得凭感觉直接下结论
2. 每个结论必须附证据（代码行 > 测试结果 > 接口返回 > 页面截图描述 > commit）
3. 证据不足时，必须明确写"证据不足"，不得自行补全
4. 发现问题后，必须检查其上下游影响，不得点到即止
5. 不允许跳过任意一项检查，若真的无法核验，必须写明"本项未核验，原因：……"

## 五、输出格式要求

### 第一部分：核验摘要

| 项目 | 内容 |
|------|------|
| 最终判定 | 通过 / 有条件通过 / 不通过 |
| 是否允许远端同步 | 是 / 否 / 有条件 |
| Bug 总数 | N 个（阻断 X / 高风险 Y / 低风险 Z） |
| 安全漏洞数 | N 个 |
| 逻辑错误数 | N 个 |
| 证据充分项 | N 项 |
| 证据不足项 | N 项 |
| 未核验项 | N 项（必须列出） |

### 第二部分：正式二次核验报告

报告必须包含以下章节，不得省略：

```
## 1. 核验范围与素材
## 2. 审核报告一致性检查（逐条）
## 3. 修复项逐条核验（7 项）
## 4. Bug 全面排查（逻辑 / 异常 / 配置 / 集成）
## 5. 安全漏洞排查
## 6. 风险残留排查
## 7. 接口全量一致性核验（28 个接口逐一列出）
## 8. 看板全量核验（5 个页面逐一列出）
## 9. 测试充分性核验
## 10. 提交与可回滚性核验（7 个 commit 逐一列出）
## 11. 残留问题汇总（分级）
## 12. 最终判定与下一步建议
```

每个章节都必须有实质内容，不得写"略"、"同上"、"参见上节"。

## 六、不允许的行为

- 不得接受已有审核结论而不独立核验
- 不得因"测试通过"就判定功能真实可用
- 不得因"页面能打开"就判定看板验收通过
- 不得因"commit 存在"就判定修复成立
- 不得跳过任意接口、页面、测试用例
- 不得省略安全漏洞排查章节
- 不得省略 Bug 主动排查章节（不是只检查已知修复项）

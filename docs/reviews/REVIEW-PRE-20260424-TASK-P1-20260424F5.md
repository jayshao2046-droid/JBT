## 三、F4 与 F5 合并训练边界

- **数据合并方式**：F4 输出 `dataset_f4.jsonl`，F5 `build_finance_corpus.py` 输出 `dataset_f5.jsonl`，由 `merge_with_f4_dataset.py` 合并去重 → `train.jsonl + val.jsonl`，统一喂入同一 LoRA 训练。
- **训练产物归属**：合并产物 `qwen3-jbt-news:14b-q4_K_M` 同时承载 F4 / F5 两个目标，模型只产出一份。
- **验收解耦**：F4 用其原 100 条新闻 holdout 验收 JSON / 品种命中 / think 链；F5 用新 100 题金融 holdout 验收金融知识；两套指标必须**全部达成**才允许 OLLAMA_MODEL 切换。
- **回滚捆绑**：F4/F5 共用一份模型文件，回滚必然同步（env 一行回切旧 qwen3:14b）。

**结论**：边界清晰，不会发生"F5 修复污染 F4 验收"的情况。

## 七、与 F4 合并签发 Token 的可行性

**结论：可行，且强烈建议合并签发**，理由：

1. F5 与 F4 共用 `train_lora_mlx.sh` / `export_to_ollama.sh`，分两次签发会出现 Token 套娃问题（同一脚本、两次 active token），不利于锁回；
2. F5 的 `merge_with_f4_dataset.py` 必须在 F4 数据准备之后立即运行，分批签发会产生交接缝隙；
3. F4/F5 模型产物只有一份、回滚捆绑，治理上视为同一变更更清晰；
4. 两单白名单已确认无路径冲突，可合并为一组 Hotfix Token 一次签发、一次锁回。

**合并签发建议**：

- Token 集合 = F4 原 4 文件 + F5 新增 4 文件，共 8 个目标。
- Token 持有方：数据 Agent（统一在 Studio 训练，Alienware 部署只走配置一行变更，走独立小批次）。
- 持有时长：建议覆盖"数据准备 → 训练 → 量化 → 推送 → A/B 第一周"，长效 Token，但禁止扩展任何新文件。

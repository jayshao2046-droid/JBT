# TASK-P1-20260425G2 — qwen3 双端再训练触发监控（Alienware researcher / decision）

## 任务类型
- P1 标准流程
- 服务归属：跨服务（services/data + services/decision）
- 设备归属：Alienware（researcher 运行侧）+ Studio（decision 运行侧）
- 当前状态：仅建档，待项目架构师预审；未冻结文件白名单，未申请 Token
- 核心原则：本任务只做“监控、评估、报警、留痕”，不自动触发重训，不自动切模型

## 背景与根因
当前 JBT 两端对 qwen3 的使用已经分化：

1. Alienware researcher 侧：qwen3 负责新闻 / 宏观 / K 线 / 研究报告输出。
2. decision 侧：qwen3 负责策略生成、代码审计、L1/L2 门控、YAML 执行器审核。

现有健康监控仅覆盖“进程是否活着、端口是否可达、报告数量、基础置信度分布”，不足以回答：

1. 当前模型质量是否正在漂移。
2. 哪一端先出现退化。
3. 是否已经达到“需要再次训练”的触发点。

因此必须建立两套相互独立的“再训练触发监控”：

1. 一套针对 Alienware researcher。
2. 一套针对 decision 策略研发链路。

二者共享同一 qwen3 家族基座，但指标体系、阈值、评测集、报警规则必须分开，避免一端表现掩盖另一端退化。

## 目标
1. 为 researcher 与 decision 建立各自独立的质量漂移监控。
2. 形成“告警级 / 复核级 / 回滚级”三层触发口径，替代人工猜测是否重训。
3. 每日沉淀运行指标，每周沉淀固定 holdout 回归评测结果。
4. 报警结果可以作为未来是否启动再训练任务的直接证据。

## 双端指标口径

### A. Alienware researcher 侧（services/data）
建议监控以下指标：

1. JSON 解析失败率。
2. 缺字段 / 多字段率。
3. think / markdown / 非 JSON 泄漏率。
4. 品种名 / 合约代码映射纠错率。
5. 决策端评分低于阈值的占比。
6. 人工复核驳回率（若有人工标签）。
7. 单条耗时、超时率、Ollama 错误率、p50/p95 延迟。

### B. decision 侧（services/decision）
建议监控以下指标：

1. `research()` 代码生成语法失败率。
2. `audit()` JSON 解析失败率与关键字段缺失率。
3. L1 / L2 门控 JSON 解析失败率。
4. L1/L2 与在线降级 / 备用模型的分歧率。
5. `yaml_signal_executor.py` 里 PASS/FAIL 审核异常率。
6. 生成后无法通过 AST / backtest / sandbox 的比例。
7. 平均耗时、超时率、fallback 率。

## 触发等级（统一框架，阈值分端设置）

### Level 1 — 告警级
- 短窗口质量指标明显劣化，但尚未形成持续趋势。
- 处理动作：发 Feishu / 邮件预警，记录 evidence，不直接重训。

### Level 2 — 复核级
- 连续多个窗口劣化，或固定 holdout 指标连续下降。
- 处理动作：自动形成“建议复核 / 建议建训练任务”的证据包。

### Level 3 — 回滚级
- 当前模型相对上一稳定版本发生明确回归。
- 处理动作：建议先回滚旧模型，再决定是否启动重训。

## 明确排除（本任务不做）
1. 不在本任务里直接训练新模型。
2. 不在本任务里自动切换线上模型。
3. 不触碰 `shared/contracts/**`、`shared/python-common/**`、`.github/**`、`docker-compose.dev.yml`。
4. 不直接修改 Mini 上任何 `services/data/**` 文件。
5. 不把两端监控混成一个统一阈值系统。

## 验收标准
1. researcher 与 decision 两端都能生成独立的每日质量指标文件。
2. researcher 与 decision 两端都能基于阈值生成 P1 / P2 级别报警。
3. 固定 holdout 评测可重复执行，结果可与上一稳定模型对比。
4. 报警内容能直接回答“为什么建议重训 / 为什么建议回滚”。
5. 全链路仍保持“人工确认后再启动训练”的治理口径。

## 建议启动时机
1. **优先级高于 decision 独立训练任务。**
2. researcher 侧建议在当前 F4/F5 新模型完成部署并稳定后立即启动，用来沉淀首个基线。
3. decision 侧建议在当前线上 qwen3 路径上先建立基线，再决定后续独立训练何时启动。

## 建议执行顺序
1. 先做 Batch A researcher 监控，建立 data 侧首个基线。
2. 再做 Batch B decision 监控，建立策略研发链路基线。
3. 最后补 Batch C 周评测与统一证据归档。
4. 待两端监控跑出稳定窗口后，再决定 G1（decision 独立训练）是否立即启动。

## 风险与回滚
- 风险：若阈值定得过严，会造成误报和频繁“建议重训”。
- 风险：若 researcher 与 decision 复用同一套阈值，容易出现误判。
- 回滚：监控任务应支持独立关闭告警，不影响业务主链运行。

## 备注
- 本任务的核心价值是把“再训练决策”变成有证据的工程动作，而不是把运维健康监控误当成模型质量监控。
- researcher 与 decision 即使未来共享同一 qwen3 家族基座，也必须长期保留两套独立触发标准。
